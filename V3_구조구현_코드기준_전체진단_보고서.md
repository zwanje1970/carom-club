# V3 구조/구현 상태 코드기준 전체진단 보고서

기준: 실제 코드 파일 기준 분석만 수행(추측/개선안 제외)

## 1) 전체 구조 요약 (한눈에)

- 라우팅의 중심은 `app`이며, 실제 운영 흐름은 `site(게시/신청)` -> `client(운영)` -> `platform(정책/승인 일부)`로 분리되어 있다.
- 루트 `/`는 `app/page.tsx`에서 `app/site/page.tsx`를 재사용하여 사이트 메인을 직접 렌더링한다.
- 인증/세션은 `v3_session` 쿠키 기반이며 `app/api/auth/session/route.ts` + `lib/auth/session.ts`에서 처리한다.
- 데이터 저장은 DB가 아니라 `lib/server/dev-store.ts`의 파일 기반 저장소(`data/v3-dev-store.json`)를 사용한다.
- `site`, `client`, `platform`에 실제 기능 페이지와 더미/구조 페이지가 혼재되어 있다(특히 `platform` 일부, `client`의 레거시 경로).

## 2) 영역별 역할 정리 (실제 경로 + 역할)

### site 영역

- `/` -> `app/page.tsx` -> `app/site/page.tsx`: 사이트 메인 재사용, 발행 카드 목록 노출.
- `/site/*` 공통 레이아웃: `app/site/layout.tsx`
  - 모바일/PC 헤더/푸터 분기, 메인 메뉴 + 테스트 진입(`/client`, `/admin`) 링크 표시.
- `/site/tournaments` -> `app/site/tournaments/page.tsx`: 전체 대회 목록 조회.
- `/site/tournaments/[id]` -> `app/site/tournaments/[id]/page.tsx`: 대회 상세 + 참가신청 진입.
- `/site/tournaments/[id]/apply` -> `app/site/tournaments/[id]/apply/page.tsx`: 신청/증빙 업로드 UI, API 저장.
- `/site/mypage` -> `app/site/mypage/page.tsx`: 내 신청 진행건 + 최근 알림.
- `/site/mypage/history` -> `app/site/mypage/history/page.tsx`: 지난 신청 기록.
- `/site/venues*`, `/site/community*`: 정적 안내/게시판 샘플 데이터 페이지.

site 역할 구현 상태: **게시/조회 중심 + 사용자 신청 진입 구현됨**.

### client 영역

- 공통 접근 제어: `app/client/layout.tsx`
  - 비로그인 -> `/login`
  - `USER` -> `/client-apply`
  - `CLIENT`라도 승인 전이면 `/client-status/*`
  - `PLATFORM`은 통과
- 핵심 실운영 경로(대회 단위):
  - `/client/tournaments` 목록, `/client/tournaments/new` 생성
  - `/client/tournaments/[id]` 운영 허브
  - `/client/tournaments/[id]/participants*` 신청자 조회/상태처리
  - `/client/tournaments/[id]/bracket*` 스냅샷/배정/확정/진행/강제수정
  - `/client/tournaments/[id]/settlement*` 정산 요약/상세
  - `/client/tournaments/[id]/card-publish` 게시카드 작성/발행
- 레거시/더미 경로도 동시 존재:
  - `/client/bracket*`, `/client/participants*`, `/client/settlement`
  - 실데이터 기반이 아닌 샘플/더미 UI가 포함됨.

client 역할 구현 상태: **실운영 경로는 입력/운영 기능이 구현됨 + 별도 레거시 더미 경로 공존**.

### admin 영역

- `/admin` -> `app/admin/page.tsx`: 관리자 전용 로그인 화면.
- `ensurePlatformAdminAccount`로 특정 플랫폼 관리자 계정 보장 후 로그인 UI 제공.
- 로그인 제출은 `app/admin/AdminLoginForm.tsx` -> `/api/auth/session` 사용 후 PLATFORM만 허용.

admin 역할 구현 상태: **플랫폼 관리자 로그인 게이트 구현됨**.

### platform 영역

- `/platform` -> `app/platform/page.tsx`: 운영/사이트 관리 진입.
- `/platform/operations/clients`: 실제 클라이언트 신청 승인/거절 동작 구현.
- `/platform/site/main-cards`: 메인 카드 템플릿 조회/수정 구현.
- `/platform/operations/users`, `/settlement`, `/support` 및 `/platform/site/pages|content|design`: 더미/구조 페이지.
- 공통 접근은 `middleware.ts`에서 `/platform/*` PLATFORM 역할로 제한.

platform 역할 구현 상태: **클라이언트 승인 + 카드 템플릿은 구현, 그 외 운영 관리 다수는 더미**.

### api 영역

- 인증: `/api/auth/signup`, `/api/auth/session` (로그인/세션조회/로그아웃)
- 신청/승인:
  - `/api/client-application`
  - `/api/platform/client-applications/[applicationId]/status`
- 대회/운영:
  - `/api/client/tournaments`, `/api/client/tournaments/[id]/participants/[entryId]/status`
  - 브래킷: snapshot/confirm/result/player/advance/조회
  - 정산: summary, details
- 게시카드:
  - `/api/platform/main-card-template`
  - `/api/client/card-template`, `/api/client/card-snapshots`
- site 신청/알림:
  - `/api/site/tournaments/[id]/apply`
  - `/api/site/notifications/[notificationId]/read`
- 이미지/OCR 연계:
  - `/api/upload/image`
  - `/api/proof-images/[imageId]`

api 역할 구현 상태: **핵심 흐름 API 대부분 구현됨**.

## 4) 구조 기준 적합성 평가

기준:
- 플랫폼 -> 틀/설정
- 클라이언트 -> 내용 입력/운영
- 사이트 -> 결과 게시

### 맞는 부분

- 플랫폼이 메인 카드 템플릿(형식/틀)을 관리하고(`app/platform/site/main-cards/page.tsx`), 클라이언트가 해당 템플릿 기반으로 카드 내용을 채워 발행(`app/client/tournaments/[id]/card-publish/page.tsx`)하며, 사이트 메인이 이를 노출(`app/site/page.tsx`)한다.
- 사이트의 대회 참가신청은 입력 수집, 실제 운영 상태 변경은 client API에서 처리되어 역할이 분리되어 있다.
- 브래킷/정산의 실운영 동작은 client 영역과 client API에 집중되어 있다.

### 어긋난 부분

- `platform`의 다수 운영 메뉴가 더미(`users/settlement/support`, `site/pages/content/design`)여서 "플랫폼=틀/정책 총괄" 역할이 일부만 실체화되어 있다.
- `site` 상세 페이지에 운영 데이터와 무관한 더미 정보 블록(상금/요강 자리 등)이 섞여 있어 "결과 게시" 관점에서 실제 데이터 게시와 더미 표기가 혼재한다.
- `client`에 실운영 경로(`/client/tournaments/[id]/*`)와 레거시 더미 경로(`/client/bracket*`, `/client/participants*`, `/client/settlement`)가 함께 존재해, "클라이언트=실운영 입력" 구조 표현이 단일하지 않다.

### 애매한 부분 (판단 필요)

- 클라이언트 신청 시점부터 세션 역할을 `CLIENT`로 갱신(`app/api/client-application/route.ts`)하지만 승인 전 접근은 `clientStatus`로 다시 차단한다.  
  - 즉 "역할"과 "승인상태"가 분리 운용되며, 역할 명칭만 보면 승인 전/후 경계가 직관적이지 않을 수 있다.

## 5) 현재 가장 위험한 구조 문제 3개

- **(1) 실운영 경로와 더미 경로 동시 존치**
  - `client` 하위에 실데이터 경로와 데모 경로가 공존하여 구조 인지/검증 시 혼선 가능성이 높다.
- **(2) 플랫폼 책임 영역의 비대칭 구현**
  - 실제 동작하는 플랫폼 기능이 "클라이언트 승인 + 카드 템플릿"에 치우쳐 있어, 플랫폼이 틀/정책을 총괄한다는 구조 목표와 구현 범위 사이 간극이 크다.
- **(3) 감사로그의 관측 경로 부재**
  - 로그는 적재되지만 조회 경로(API/UI)가 없어 운영자가 실제 추적/검증에 활용하기 어렵다(저장만 되는 상태).
