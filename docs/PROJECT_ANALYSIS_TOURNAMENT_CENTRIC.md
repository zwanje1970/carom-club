**목표**: 서비스를 커뮤니티/랭킹 중심이 아니라 **대회 홍보 / 참가자 모집 / 참가자 관리 / 대진표 생성 / 대회 운영** 중심으로 재정리하기 위한 현황 분석.

## 2. 현재 기능 구조

### 2.1 대회 생성

- **구현 위치**: `/client/tournaments/new`, `/admin/tournaments` POST.
- **내용**: 이름, 일시, 장소(venue), 상태, 참가비, 최대 인원, 참가 조건, 경기 방식, 상금, 요강, 홍보 내용, **경기장 수·경기장별 이름/주소/전화** (client new·edit).
- **상태**: 이미 구현됨. 클라이언트가 새 대회 만들기 → 기본정보·경기장 슬롯 입력 후 저장.

### 2.2 대회 상세

- **사용자**: `/tournaments/[id]` — 안내, 요강, 참가신청, 참가자 명단, 문의, 결과 탭. 경기장(matchVenues) 표시, 대진·결과 링크, **대진표 알림 받기** 버튼.
- **클라이언트**: `/client/tournaments/[id]` — 기본정보, 대회요강, 참가자, 경기장(zones), 대진표, 결과, 공동관리자, 홍보페이지. 상태·일시·경기장 블록 표시.
- **관리자**: `/admin/tournaments/[id]` — 조회·PATCH(상태 등).
- **상태**: 이미 구현됨. 공개 상세와 클라이언트 운영 상세가 분리되어 있음.

### 2.3 참가 신청

- **사용자**: `/tournaments/[id]` 참가신청 탭 → `TournamentApplyForm` → POST `/api/tournaments/apply`. 입금자명, 소속, 동의. 신청 완료 후 **대진표 알림 받기** 노출.
- **API**: 참가비·정원·대기 여부 등 검증, APPLIED 생성, 알림 메시지.
- **상태**: 이미 구현됨. 참가 취소(entry/cancel)도 있음.

### 2.4 참가자 관리

- **클라이언트**: `/client/tournaments/[id]/participants` — 참가자 테이블, 확정/반려, 권역 배정(다권역 시). 기능 코드 `PARTICIPANT_MANAGEMENT`로 제한.
- **API**: confirm, reject, attendance, absent 등. BRACKET_GENERATED 이후 확정/반려 잠금.
- **상태**: 이미 구현됨. 다권역 배정은 연맹 대회용으로 구조만 있음.

### 2.5 대진표 관련

- **단판 토너먼트**:  
  - 클라이언트: `/client/tournaments/[id]/bracket` — 경기장 표시, **대진표 생성** 버튼. CLOSED 이후만 생성, CONFIRMED만 대상, 생성 후 BRACKET_GENERATED.  
  - API: POST `.../bracket/generate` (type=tournament) → TournamentRound + bracketData, status=BRACKET_GENERATED, 참가자 푸시.
- **권역/본선**:  
  - 권역별 대진(TournamentZoneMatch), 본선(TournamentFinalMatch), 진출자 취합 등 API·UI 존재.  
  - 연맹 대회용으로 복잡도 높음.
- **공개**: `/tournaments/[id]/bracket` — 본선 대진표 관람.
- **상태**: 단판 토너먼트는 이미 구현·운영 가능. 권역/본선은 구조 있으나 연맹 전용, 당구장 MVP와는 분리된 흐름.

### 2.6 당구장 홍보 관련

- **당구장 목록/상세**: `/venues`, `/v/[slug]` — 승인된 VENUE 조직 목록, slug로 상세(설명, 연락처, 테이블/요금 등).
- **대회 홍보**:  
  - 클라이언트: `/client/tournaments/[id]/promo` — promoContent 편집·미리보기. 기능 코드 `TOURNAMENT_PROMO_PAGE`.  
  - 공개: 대회 상세에서 요강·홍보 내용 노출.
- **상태**: 당구장 홍보 페이지는 구현됨. 대회 홍보는 편집·노출 구현, 공유용 단일 URL 등은 보강 여지 있음.

### 2.7 알림 관련

- **사이트 내 알림**: `Notification` 테이블, `/api/notifications`, `/notifications-popup`. 참가 확정 등 메시지 저장·읽음 처리.
- **Web Push**:  
  - 구독: POST `/api/push/subscribe`, 해제: POST `/api/push/unsubscribe`, VAPID: GET `/api/push/vapid-public`.  
  - 이벤트: 참가 승인, 대진표 생성, 경기장 안내(12h 전 cron), 빵빠레(우승/준우승/준결승).  
  - Service Worker `/sw.js`, **대진표 알림 받기** 버튼(대회 상세·참가 완료 화면).
- **상태**: 이미 구현됨. 경기 호출 알림은 미구현(요구사항에서 제외).

## 3. 기능 구분 (구현 수준·우선순위)

### 3.1 이미 구현되어 있는 기능

- 대회 생성(클라이언트/관리자), 기본정보·경기장 슬롯 입력
- 대회 상세(공개·클라이언트·관리자)
- 참가 신청·취소
- 참가자 목록·확정/반려, BRACKET_GENERATED 후 잠금
- 단판 토너먼트 대진표 생성(CLOSED 후, CONFIRMED 대상, BRACKET_GENERATED 전환)
- 대회 요강(초안/공개)·홍보 콘텐츠 편집 및 노출
- 당구장 목록/상세(v/slug)
- Web Push 4종(참가 승인, 대진표 생성, 경기장 안내, 빵빠레)
- 사이트 내 알림(읽음 처리)
- 클라이언트 신청·승인, 업체 설정, 기능 권한(연회원/기능 코드)

### 3.2 부분적으로 구현된 기능

- **결과 관리**: 권역 결과·본선 결과 입력/표시 구조 있음. 단판 토너먼트의 “승자 입력 → 다음 라운드” 연동은 TournamentRound bracketData 기반으로 보강 필요할 수 있음.
- **대회 상태 흐름**: DRAFT → OPEN → CLOSED → BRACKET_GENERATED → FINISHED 는 반영됨. UI/가이드(상태별 다음 액션)는 더 명확히 할 여지 있음.
- **공동관리자/권역 관리자**: 배정·권한 구조는 있으나, “공동관리자만 보는 메뉴” 등 UX는 단순 안내 수준.

### 3.3 구조는 있으나 실제 동작이 약한 기능

- **커뮤니티** (`/community`): 제목/부제만 있고 게시판·랭킹 등 실사용 기능 없음.
- **권역/연맹 대회**: TournamentZone, 권역별 대진·진출자 취합·본선 대진 API/UI 존재하나, 당구장 MVP와는 별도. “경기장” 탭은 당구장 MVP용으로 쓰이고, 권역은 연맹 전용.
- **회원/랭킹**: 회원 목록(관리자), 멤버 프로필(핸디/AVG)은 있으나, 랭킹·리그 기능은 없음.
- **정산/결과 시스템**: SETTLEMENT_SYSTEM 기능 코드로 결과 관리 접근 제한. 정산·상금 지급 등 실제 정산 플로우는 미구현.

### 3.4 불필요하거나 현재 우선순위에서 뒤로 미뤄야 할 기능

- **커뮤니티**: 대회 중심 재정리 목표에서는 우선순위 낮음. 유지하되 “준비 중” 수준으로 두거나, 대회/당구장 콘텐츠로 대체 검토.
- **권역/연맹 전체 플로우**: 당구장 대회 MVP와 무관. 연맹 전용으로 문서화·분리하고, 당구장 중심 재정리 시에는 뒤로 미룸.
- **플랫폼 관리자 대회 실무**: 대시보드에서도 “대회 실무는 클라이언트에서” 안내. 대회 생성/참가자/대진표는 클라이언트가 주체. 관리자는 현황·설정 위주로 유지.
- **회비 장부·등록상품·요금제 상세**: 플랫폼 수익화용. 당구장 대회 운영만 놓고 보면 후순위.

### 3.5 새로 추가하거나 강화해야 할 기능

- **대회 홍보**:  
  - 공유용 단일 URL(예: `/tournaments/[id]/promo` 공개 페이지) 정리, SNS/카카오 공유 등 노출·공유 강화.
- **참가자 모집**:  
  - 마감 임박·정원 임박 등 사용자 노출, 검색/필터(날짜, 지역, 상태) 보강.
- **대회 운영**:  
  - 상태별 “다음에 할 일” 가이드(예: 참가 마감 → 대진표 생성 → 결과 입력).  
  - 단판 토너먼트 결과 입력 후 “승자 반영”이 대진표/결과 화면에 명확히 반영되도록.
- **경기장 안내**:  
  - 이미 12시간 전 푸시·경기장 블록 표시 있음. 필요 시 지도/길찾기 등은 후순위로.

## 4. 핵심 목표별 분석

### 4.1 대회 홍보

- **현재**: 대회 상세(공개), 요강·홍보 콘텐츠, 경기장 블록, 대진·결과 링크. 클라이언트는 홍보 탭에서 편집. 당구장 상세(`/v/[slug]`)로 당구장 자체 홍보.
- **부족한 점**: “이 대회만 링크해서 보내기”용 홍보 전용 URL이 사용자 메뉴/공유 플로우에 드러나지 않음. SNS·카카오 공유 버튼 등 없음.
- **우선순위**: 중. 공개 대회 상세를 그대로 홍보용으로 쓰거나, `/tournaments/[id]`를 공유용으로 명확히 하고, 공유 CTA 추가.

### 4.2 참가자 모집

- **현재**: 대회 목록(공개), 상세, 참가 신청 폼, 정원/대기 제한, 참가 취소. 알림(참가 승인, 대진표 생성 등)으로 재방문 유도.
- **부족한 점**: “모집 중” 강조, 마감 임박/정원 임박 배지, 날짜·지역·상태 필터 등이 약함. 목록이 단순 리스트에 가깝다.
- **우선순위**: 중상. 목록·상세를 “모집” 관점으로 재구성(상태·일시·정원 노출, CTA 강화).

### 4.3 참가자 관리

- **현재**: 클라이언트 대회별 참가자 탭, 확정/반려, 출석/불참, 권역 배정(연맹). BRACKET_GENERATED 후 확정/반려 잠금. 대시보드에 “참가 승인 대기” 수 노출.
- **부족한 점**: 대기 순번 변경, 일괄 확정 등은 없음. 당구장 MVP 기준으로는 충분할 수 있음.
- **우선순위**: 유지·소소한 개선. 대회 중심 재정리 시 “참가자 관리”는 이미 핵심 경로에 있음.

### 4.4 대진표 생성

- **현재**: 단판 토너먼트 — CLOSED 후 생성, CONFIRMED만 대상, TournamentRound+bracketData, BRACKET_GENERATED 전환, 푸시. 경기장 블록 표시.
- **부족한 점**: 단판 결과 입력 후 “승자 → 다음 라운드” 자동 반영이 UI/데이터 모델에서 완전히 연결되었는지 확인 필요. 권역/본선은 연맹용으로 유지.
- **우선순위**: 높음. 당구장 MVP의 핵심. 단판 플로우(생성 → 결과 입력 → 진출) 안정화 및 문서화.

### 4.5 대회 운영

- **현재**: 클라이언트 대시보드(접수중/마감/종료, 승인 대기, 새 대회·목록·참가자·대진표 링크). 대회별 탭(기본정보, 요강, 참가자, 경기장, 대진표, 결과, 공동관리자, 홍보). 상태값(DRAFT/OPEN/CLOSED/BRACKET_GENERATED/FINISHED) 반영.
- **부족한 점**: “지금 해야 할 일”이 단계별로 안내되지 않음(예: 참가 마감 전 → 마감 처리 → 대진표 생성 → 결과 입력). 결과 입력 후 “대회 종료” 처리와 빵빠레 푸시는 연동됨.
- **우선순위**: 높음. 상태 기반 운영 가이드(체크리스트/바로가기) 추가하면 대회 중심 재정리에 잘 맞음.

## 5. 문제점 요약

1. **정보 구조**: 사용자 메뉴가 홈/당구장/대회/커뮤니티/마이페이지로 되어 있어, “대회 보기·참가하기”가 메인인지 한눈에 안 들어옴.
2. **이중 진입점**: 대회 실무가 `/admin/tournaments`와 `/client/tournaments`에 나뉘어 있고, 관리자 대시보드는 “클라이언트에서 하라”고 안내. 신규 클라이언트는 어디로 가야 할지 헷갈릴 수 있음.
3. **기능 권한**: 참가자 관리·대진표·요강·홍보·결과 등이 기능 코드로 막혀 있어, 연회원/일반 업체에 따라 화면이 갈림. 문서화·온보딩이 없으면 “왜 이 메뉴가 안 보이지?”가 발생.
4. **커뮤니티/권역**: 커뮤니티는 실질 기능 없음. 권역은 연맹용으로 복잡. 당구장 MVP와 혼재되어 “대회 하나 만들고 참가 받고 대진표 돌리는” 스토리가 약해 보일 수 있음.
5. **대회 목록**: 공개·클라이언트 목록이 “모집 중” 강조, 필터, 정렬이 약해 “참가자 모집” 느낌이 약함.

## 6. 개선 우선순위 제안

**1순위 (대회 운영·대진표 안정화)**  
- 단판 토너먼트: 생성 → 결과 입력 → 승자 반영 → 대회 종료(빵빠레) 플로우 검증 및 필요 시 보강.  
- 클라이언트 대회 상세에 “상태별 다음 액션” 가이드(체크리스트/바로가기) 추가.  
- 대회 실무 진입은 **클라이언트 대시보드 → 대회** 로 일원화하고, 관리자 대회 메뉴는 “전체 현황·예외 조치” 용도로 명시.

**2순위 (참가자 모집·홍보)**  
- 대회 목록: “모집중” 강조, 마감/정원 임박 표시, 날짜·상태 필터.  
- 대회 상세: 공유용 URL 안내, SNS/카카오 공유 버튼 등(선택).  
- 대회 홍보 탭: 공개 URL 정리(예: `/tournaments/[id]` 또는 전용 promo 경로).

**3순위 (구조 정리·문서)**  
- 네비게이션/홈: “대회”를 더 앞에 두거나, 홈 첫 블록을 “진행 중인 대회” 등으로 재구성.  
- 기능 권한(연회원/기능 코드) 정리 문서 및 클라이언트 온보딩(첫 로그인 시 “대회 만들기 → 참가 받기 → 대진표” 안내).  
- 연맹/권역 기능은 “연맹 대회 가이드”로 분리하고, 당구장 대회 플로우와 구분해 문서화.

**4순위 (후순위)**  
- 커뮤니티: 유지하되 “준비 중” 또는 대회/당구장 콘텐츠로 대체 검토.  
- 랭킹·리그: 대회 중심 재정리 범위 밖.  
- 회비 장부·등록상품·요금제 상세: 플랫폼 수익화 단계에서 진행.
