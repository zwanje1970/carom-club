# 작업 원칙

## 1. 우선순위

- **현재 우선순위는 커뮤니티/랭킹/자유게시판이 아니라 대회 운영 완성도다.**

## 2. 구현 우선순위 (1 → 10)

| 순위 | 항목 | 비고 |
|------|------|------|
| 1 | 대회 생성/수정 | |
| 2 | 대회 상세 홍보페이지 | |
| 3 | 참가 신청 + 입금완료 체크 + 관리자 입금확인 + 자동 참가확정/대기자 처리 | |
| 4 | 참가자 관리 화면 | |
| 5 | 대기자 자동승격 | |
| 6 | 대진표 생성 | |
| 7 | 대진표 강제수정 | |
| 8 | 대회 카드/목록 참가현황 표시 | |
| 9 | 위치기반 당구장 노출 | |
| 10 | 향후 유료화 대비 필드 반영 | |

## 3. 작업 순서 (한 번에 전부 덮어쓰지 말 것)

1. **기존 구조 분석** — 관련 코드·DB·API·페이지 파악
2. **페이지 구조 / DB 구조 / 상태값 / 권한 먼저 정리** — 문서화 후 설계 일치 여부 확인
3. **그다음 실제 구현** — 정리된 범위 안에서만 수정·추가

## 4. UX 원칙

- **모바일 사용성 우선** 고려
- **실제 당구장 운영자**가 **버튼 수 적게**, **빠르게** 처리할 수 있게 설계
- 불필요한 단계·화면·입력은 추가하지 말 것

구체적인 대회 운영 흐름·페이지·DB·상태값·권한은 `TOURNAMENT_OPERATION_STRUCTURE.md` 참고.
